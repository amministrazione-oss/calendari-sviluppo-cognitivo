## Language

Rispondi sempre in italiano, indipendentemente dalla lingua usata nel prompt.

## What this repo is

A single-file client-side web app (`index.html`, ~2800 righe al 20/07/2026: valore indicativo ‚Äî il conteggio esatto si verifica con `wc -l` solo durante gli audit; inline `<style>` + inline `<script>`, no build step) for **Centro Sviluppo Cognitivo**, a cognitive-training center (BrainRx / Metodo Feuerstein) running sites in Cesate, Busto Arsizio, online and at-home. It's a scheduling/CRM tool ("Gestionale") for managing operators (therapists), clients ("utenti"), projects/treatment plans, and calendar sessions. UI and identifiers are in Italian.

There is no package.json, build tool, linter, or test suite ‚Äî this is intentionally a plain HTML/CSS/JS deploy. `CNAME` configures GitHub Pages for the custom domain `calendari-sviluppo-cognitivo.it`; pushing to `main` is the deploy.

## Running / testing changes

- Opening `index.html` directly as a `file://` URL is explicitly blocked by the app (see the `init()` guard) ‚Äî it will show an error screen.
- Full functionality (login) requires being served over `https://` from an origin registered as a redirect URI in the Entra ID (Azure AD) app registration referenced by `CFG.clientId`/`CFG.tenantId`. In practice this means auth only works on the deployed production domain (or another origin explicitly added to the app registration) ‚Äî there's no local dev auth flow.
- You can still statically inspect/edit markup, CSS, and non-auth-dependent JS logic (e.g. the scheduling algorithm in `generateMonth`) without deploying; just be aware MSAL/Graph calls won't succeed off the registered domain.
- No automated tests exist. Verify changes by reasoning through the code paths and, where feasible, deploying to a real/staging domain and exercising the UI manually.
- **Node.js v24 (LTS) √® installato sul PC dal 17/07/2026** (in precedenza assente ‚Äî vedi `CONTESTO.md` per lo storico). In ogni ciclo che modifica `index.html`, il controllo sintattico √® **obbligatorio con un parser reale** (`node --check` sui blocchi `<script>` estratti, o equivalente) ‚Äî non basta pi√π il solo bilanciamento di parentesi/graffe/quadre usato nei cicli precedenti a questa data.
- **`check-sintassi.js`** (radice del repo, da Ciclo C, 17/07/2026) √® lo strumento ufficiale di collaudo automatico ‚Äî sostituisce gli script temporanei riscritti a ogni ciclo nei cicli B/B.1: un solo comando, `node check-sintassi.js`. ## Architecture

Everything lives in `index.html`: CSS in one `<style>` block, then several `<script>` blocks. Read top-to-bottom, the script is organized into:

1. **MSAL loader** ‚Äî tries a list of CDN URLs/versions for `msal-browser.js` with fallback (`tryNext()`), then dispatches `msal-ready`.
2. **`CFG` constant** (near the top of the main script) ‚Äî hardcodes the Entra ID `clientId`/`tenantId`, the SharePoint host (`spHost`), the SharePoint **list names** used as the "database" (`CFG.lists`), a bootstrap `adminEmails` list, and `claudeProxy` (an external HTTPS endpoint, `https://csc-claude-proxy.azurewebsites.net/api/claude-proxy`). The client always speaks the Anthropic Messages API shape both ways ‚Äî requests as `{model, max_tokens, messages:[{role,content}]}` (`generateMonthAI`, `sendChat`) and responses read back as `data.content` filtered to `type==='text'` ‚Äî but the proxy currently translates these server-side to **Gemini 2.0 Flash** rather than actually calling Anthropic ‚Äî keep the client-facing format in mind if this changes (vedi `CONTESTO.md`, "Storico decisioni e implementazioni", per lo stato della decisione vendor/fatturazione). No API key of any kind is present in this file ‚Äî whichever key the proxy uses lives server-side behind it. Changing tenants/sites means updating this object.
3. **Data layer = SharePoint Lists via Microsoft Graph** (`gfetch`, `resolveSite`, `tryResolveLists`, `loadListRecords`, `loadAll`, `saveRecord`, `deleteRecord`). There is no real backend/database: each SharePoint list item stores one entity as a JSON blob in a `Data` column (`Title` = the entity's own `id`). `CFG.lists` names seven lists; six are required for `tryResolveLists()` to succeed (`operatori`, `utenti`, `progetti`, `sessioni`, `chiusure`, `impostazioni`) and one, `report` (S3, Ciclo E, 20/07), is listed in `CFG.listeOpzionali` ‚Äî if that list doesn't exist yet on SharePoint, resolution still succeeds and the app runs normally, only the "Report precedenti" history stays unavailable (`renderReportStorico()` shows a message instead of failing). `loadAll()` pulls all resolved lists into `state.data` and also runs five one-time migrations (corrected here ‚Äî this bullet had fallen behind the code, only described the first two): splitting legacy `nome` fields into `nome`+`cognome`; (17/07, Ciclo A) rewriting legacy project `sede` values `Presenza+Online`/`Presenza+Domicilio` to `Cesate+Online`/`Cesate+Domicilio`; (17/07, Ciclo B/S6) converting `monteOre`/`metodi[].oreTotali` from decimal hours to minutes (marker `p.oreInMinuti`); (17/07, Ciclo B/S8) splitting operator `tempoBusto` into `tempoBustoCesate`/`tempoBustoCasa`; (17/07, Ciclo D/S9) consolidating any legacy session `esito` field into `stato` (defensive ‚Äî no such field was ever actually written by this code, see `CONTESTO.md`, Registro delle decisioni). `saveRecord(key,record)` treats every list as an array of entities in `state.data[key]` **except** the two named in `LISTE_RECORD_SINGOLO`/`isRecordSingolo()` (`chiusure`, `impostazioni`, which store one settings-like object instead) ‚Äî an explicit, fixed list (Ciclo E.2, 20/07; previously this was inferred from `Array.isArray(state.data[key])` at call time, which depended on how/when that key had been populated rather than on what the list actually is).
4. **Global mutable `state` object** ‚Äî holds the MSAL account, resolved `role`, current user (`me`), SharePoint `siteId`/`listIds`, and `state.data.{operatori,utenti,progetti,sessioni,chiusure,impostazioni,report}`. There's no framework/store; UI code reads/writes `state` directly and re-renders by calling the relevant `renderX()` function.
5. **Auth + role resolution** (`initMsal`, `doLogin`, `doLogout`, `getToken`, `resolveRole`) ‚Äî role is derived from Entra ID app roles claim, or by matching the signed-in email against the `operatori` list, or against `CFG.adminEmails`. If no operators exist yet, the first login bootstraps as Admin (`state.bootstrap`). Two roles: `Admin` (full access) and `Operatore` (sees only their own calendar/sessions/availability).
6. **Navigation** (`TABS`, `buildNav`, `showTab`) ‚Äî each tab maps to a `<section id="view-*">` and triggers a `renderX()` call; there's no router, just show/hide of sections. A `TABS` entry may carry a `children` array instead of being a direct leaf: `buildNav()` renders it as a dropdown (`.nav-drop`/`.nav-drop-toggle`/`.nav-drop-menu`) opened by hover on pointer devices and by click/tap on the parent button otherwise (`chiudiNavDrop()`, a document-level click listener) ‚Äî needed since touch has no hover. Only Admin's "‚öôÔ∏è Impostazioni" entry does this today (children: Impostazioni itself + "üìÖ Chiusure", Ciclo H, 21/07) ‚Äî Chiusure has no top-level nav entry of its own anymore, only reachable through that dropdown. `trovaTabById(role,id)` looks up a tab (leaf or nested child) by id; `showTab()` marks the parent toggle `.active` too whenever the currently shown id is one of its children, so the dropdown's origin stays visible.
7. **Rendering** ‚Äî plain DOM/innerHTML templating per view (`renderCalendario`, `renderSessioni`, `renderOperatori`, `renderUtenti`, `renderProgetti`, `renderDisponibilita`, `renderChiusure`, `renderImpostazioni`, ...), all string-built HTML with the `esc()` helper for escaping. Modals are generic: `openModal(html, onCancel, isDirty, onSaveTrigger)`/`openModalStacked(html, onCancel, isDirty, onSaveTrigger)` inject markup into `#modal-root`; `onCancel` restores state on abandonment (e.g. the disponibilit√Ý snapshot), `isDirty`/`onSaveTrigger` power the unsaved-changes guard (see "Regole UX" below). ### Key domain concepts worth knowing before editing scheduling/session code

- **Sedi** (sites): `Cesate`, `Busto Arsizio`, `Online`, `Domicilio`, plus composite modes `Cesate+Online` / `Cesate+Domicilio` (rinominate dal 17/07, Ciclo A ‚Äî erano `Presenza+Online`/`Presenza+Domicilio`; vedi `CONTESTO.md`, Registro delle decisioni, per data e motivazione) that resolve to a concrete site per-session based on operator/room availability. **Business rule**: for `Cesate+Online` the effective site can only resolve to `Cesate` or `Online`; for `Cesate+Domicilio` only to `Cesate` or `Domicilio` ‚Äî **never** `Busto Arsizio` (Busto Arsizio has no online sessions). This is enforced in `generateMonth()` (slot-compatibility filter and effective-`sede` resolution) and in `generateMonthAI()` (prompt constraint **and** a hard post-validation that discards any AI-proposed session violating it) ‚Äî keep both in sync if this rule ever changes.
- **Aule** (rooms): fixed lists `AULE_CESATE` (6) and `AULE_BUSTO` (2); an operator keeps one room for the whole day/half-day once assigned (`opRoom` map keyed by `operatorId|date` or `operatorId|date|halfDay`). An `Online` session resolved as "in sede" (see Passata 2 above) still occupies a room like an in-presence one ‚Äî it does **not** free the room.
- **Orari e giorni**: the center's bookable window is **09:00‚Äì19:30**, enforced everywhere a slot is computed (`tmin('09:00')`/`tmin('19:30')` clamps in Passata 1, Passata 3, and the AI prompt's "VINCOLI INDEROGABILI"). **Sunday is always excluded from generation** ‚Äî every day-loop in `generateMonth`/Passata 3 skips `dn==='Dom'` outright, and the AI prompt states "Domenica: centro chiuso".
- **Disponibilit√Ý** (availability): weekly recurring slots plus per-date exceptions (`eccezioni`/`availOverrides`), read through `effRng()`, which supports both a newer array-of-exception-objects format and a legacy per-date object format ‚Äî check both when touching availability logic. Each weekly/exception slot can carry a `sedi` list (`sdComp`/the site-compatibility check around line 1541): a slot tagged only `Cesate` does **not** implicitly cover `Online` (or vice versa) ‚Äî for a composite `Cesate+Online`/`Cesate+Domicilio` project the slot must explicitly list `Cesate` **or** the remote site to match. **S5 (Ciclo C, 17/07)**: in the project editor (weekly `pe-av`, daily exception fasce, and "Applica a pi√π giorni"), the sede tags offered for a slot are restricted to `sediAmmesseProgetto(p.sede)` ‚Äî the project's own `Sede` (single site ‚Üí that tag; composite ‚Üí Cesate + the remote one). A tag already selected on an existing slot is never hidden/force-unchecked even if it falls outside the newly-allowed set after a `Sede` change mid-edit ‚Äî it stays visible, checked, and visually flagged (`sede-extra` class) alongside an explicit warning banner listing the affected days, so nothing is silently dropped.
- **Assenze operatore** (via `eccezioni`, read by `effRng`): `malattia` always blocks the **entire day**, no time-band option. `ferie`/`permesso_visita`/`permesso_studio` take an **optional** time band (`da`/`a`): with no band, the whole day is unavailable; with a band, only that band is subtracted from the operator's own declared weekly availability for that day (`subtractWindow(weekly(), da, a)`) ‚Äî the rest of the day stays available **only inside the operator's already-declared weekly availability slots for that day, never beyond them** (the exception can shrink availability, it can never grant more than what's on file).
- **Requisito formazioni**: when a project has `tipiSessione`, an operator is only eligible for a given rotation slot if they hold **every** training (`formazione`) required by that session type's `componenti` (`reqForms.every(rf=>opForms.includes(rf))`, Passata 1 and Passata 3) ‚Äî a partial match is not enough.
- **Tipi di sessione** (session-type rotation): when a project has `tipiSessione`, the rotation index (`sessionCountByProject`, keyed per project) advances continuously across the **entire month** and is never reset at the start of a week ‚Äî including for sessions recovered by Passata 3, which keep incrementing the same counter. **Approccio A**: one single operator covers the whole session (all of that session type's method "componenti"), never split across multiple operators.
- **Progetti** (projects) carry `monteOre` (hour budget) ‚Äî projects flagged `noMonteOre` (manual/no-hour-budget scheduling) are **excluded from automatic generation entirely** (filtered out of scope at both `determinaAmbito` call sites) and must be calendared by hand ‚Äî `modalitaFrequenza` (`settimanale`/`quindicinale`/`mensile`, Ciclo F.1, 21/07 ‚Äî see "Modifiche dal campo", S11), `frequenza` (sessions/week, used only in `settimanale` mode; `null` for the other two), `durataSessione`, `operatoriAmmessi` (eligible operators), an optional `operatoreFisso` flag (already implemented: when set, placement only ever tries the *first* operator in `operatoriAmmessi`, never rotating to others), and optionally `tipiSessione` (session-type rotations with per-method duration "componenti").
- **Sessioni** states: `proposta`, `confermata`, `eseguita`, `assenza ingiustificata`, `annullata` (see `STATI_SESS`). Only in-scope `proposta` sessions of origin `generata` are ever deleted/recreated by a generation run or by "Svuota proposte del mese" ‚Äî every other state, plus any `proposta` of origin `manuale` (see Scheduling engine, "Origine `manuale` vs `generata`"), is a protected, active scheduling constraint. `oreErog()` (monte-ore consumed) counts `eseguita` and `assenza ingiustificata`, excludes `annullata` (which stays recoverable) and `proposta`/`confermata` (not yet consumed). Sessions from **different projects of the same client never overlap**: every placement/repair pass collects busy-blocks across *all* of that client's projects (`allPU`/`uBN`, not just the project being scheduled) and requires the candidate slot to be free against that combined set before placing.
- **Chiusure**: center-wide closures (holidays/vacation) that block session generation on those dates; Italian public holidays (including computed Easter/Easter Monday) are precomputed via `getFestivita`/`getPasqua`.

When making changes, keep everything self-contained in `index.html` (no framework or bundler is set up), preserve the JSON-blob-in-SharePoint-list persistence model, and be careful with the two scheduling code paths (`generateMonth` vs `generateMonthAI`) since they duplicate validation/session-shape logic independently.
