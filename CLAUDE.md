# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

> **Storico**: questo file contiene solo regole invarianti e istruzioni operative. Per decisioni passate con motivazione, cronologia delle implementazioni, bug risolti e discussioni chiuse, vedi `CONTESTO.md` (sezioni "Registro delle decisioni" e "Storico decisioni e implementazioni").

## Language

Rispondi sempre in italiano, indipendentemente dalla lingua usata nel prompt.

## What this repo is

A single-file client-side web app (`index.html`, ~2800 righe al 20/07/2026: valore indicativo — il conteggio esatto si verifica con `wc -l` solo durante gli audit; inline `<style>` + inline `<script>`, no build step) for **Centro Sviluppo Cognitivo**, a cognitive-training center (BrainRx / Metodo Feuerstein) running sites in Cesate, Busto Arsizio, online and at-home. It's a scheduling/CRM tool ("Gestionale") for managing operators (therapists), clients ("utenti"), projects/treatment plans, and calendar sessions. UI and identifiers are in Italian.

There is no package.json, build tool, linter, or test suite — this is intentionally a plain HTML/CSS/JS deploy. `CNAME` configures GitHub Pages for the custom domain `calendari-sviluppo-cognitivo.it`; pushing to `main` is the deploy.

## Running / testing changes

- Opening `index.html` directly as a `file://` URL is explicitly blocked by the app (see the `init()` guard) — it will show an error screen.
- Full functionality (login) requires being served over `https://` from an origin registered as a redirect URI in the Entra ID (Azure AD) app registration referenced by `CFG.clientId`/`CFG.tenantId`. In practice this means auth only works on the deployed production domain (or another origin explicitly added to the app registration) — there's no local dev auth flow.
- You can still statically inspect/edit markup, CSS, and non-auth-dependent JS logic (e.g. the scheduling algorithm in `generateMonth`) without deploying; just be aware MSAL/Graph calls won't succeed off the registered domain.
- No automated tests exist. Verify changes by reasoning through the code paths and, where feasible, deploying to a real/staging domain and exercising the UI manually.
- **Node.js v24 (LTS) è installato sul PC dal 17/07/2026** (in precedenza assente — vedi `CONTESTO.md` per lo storico). In ogni ciclo che modifica `index.html`, il controllo sintattico è **obbligatorio con un parser reale** (`node --check` sui blocchi `<script>` estratti, o equivalente) — non basta più il solo bilanciamento di parentesi/graffe/quadre usato nei cicli precedenti a questa data.
- **`check-sintassi.js`** (radice del repo, da Ciclo C, 17/07/2026) è lo strumento ufficiale di collaudo automatico — sostituisce gli script temporanei riscritti a ogni ciclo nei cicli B/B.1: un solo comando, `node check-sintassi.js`. Esegue `node --check` su ogni blocco `<script>` di `index.html` e, in coda, una batteria di test funzionali sulle funzioni pure toccate dai cicli di lavoro (`parseHM`/`fmtHM`, `tempoBustoOperatore`, `decidiOnlineDaCasa`, ...), estratte **direttamente dal file reale** (non da una copia ritrascritta) tramite le liste `EXTRACT_FUNZIONI`/`EXTRACT_COSTANTI` in testa al file. **Da eseguire prima di ogni commit che tocca `index.html`**; quando un ciclo tocca una nuova funzione pura, aggiungerla a quelle liste ed estendere la sezione test con i casi rilevanti (non riscrivere uno script a parte).

## Architecture

Everything lives in `index.html`: CSS in one `<style>` block, then several `<script>` blocks. Read top-to-bottom, the script is organized into:

1. **MSAL loader** — tries a list of CDN URLs/versions for `msal-browser.js` with fallback (`tryNext()`), then dispatches `msal-ready`.
2. **`CFG` constant** (near the top of the main script) — hardcodes the Entra ID `clientId`/`tenantId`, the SharePoint host (`spHost`), the SharePoint **list names** used as the "database" (`CFG.lists`), a bootstrap `adminEmails` list, and `claudeProxy` (an external HTTPS endpoint, `https://csc-claude-proxy.azurewebsites.net/api/claude-proxy`). The client always speaks the Anthropic Messages API shape both ways — requests as `{model, max_tokens, messages:[{role,content}]}` (`generateMonthAI`, `sendChat`) and responses read back as `data.content` filtered to `type==='text'` — but the proxy currently translates these server-side to **Gemini 2.0 Flash** rather than actually calling Anthropic — keep the client-facing format in mind if this changes (vedi `CONTESTO.md`, "Storico decisioni e implementazioni", per lo stato della decisione vendor/fatturazione). No API key of any kind is present in this file — whichever key the proxy uses lives server-side behind it. Changing tenants/sites means updating this object.
3. **Data layer = SharePoint Lists via Microsoft Graph** (`gfetch`, `resolveSite`, `tryResolveLists`, `loadListRecords`, `loadAll`, `saveRecord`, `deleteRecord`). There is no real backend/database: each SharePoint list item stores one entity as a JSON blob in a `Data` column (`Title` = the entity's own `id`). `CFG.lists` names seven lists; six are required for `tryResolveLists()` to succeed (`operatori`, `utenti`, `progetti`, `sessioni`, `chiusure`, `impostazioni`) and one, `report` (S3, Ciclo E, 20/07), is listed in `CFG.listeOpzionali` — if that list doesn't exist yet on SharePoint, resolution still succeeds and the app runs normally, only the "Report precedenti" history stays unavailable (`renderReportStorico()` shows a message instead of failing). `loadAll()` pulls all resolved lists into `state.data` and also runs five one-time migrations (corrected here — this bullet had fallen behind the code, only described the first two): splitting legacy `nome` fields into `nome`+`cognome`; (17/07, Ciclo A) rewriting legacy project `sede` values `Presenza+Online`/`Presenza+Domicilio` to `Cesate+Online`/`Cesate+Domicilio`; (17/07, Ciclo B/S6) converting `monteOre`/`metodi[].oreTotali` from decimal hours to minutes (marker `p.oreInMinuti`); (17/07, Ciclo B/S8) splitting operator `tempoBusto` into `tempoBustoCesate`/`tempoBustoCasa`; (17/07, Ciclo D/S9) consolidating any legacy session `esito` field into `stato` (defensive — no such field was ever actually written by this code, see `CONTESTO.md`, Registro delle decisioni). `saveRecord(key,record)` treats every list as an array of entities in `state.data[key]` **except** the two named in `LISTE_RECORD_SINGOLO`/`isRecordSingolo()` (`chiusure`, `impostazioni`, which store one settings-like object instead) — an explicit, fixed list (Ciclo E.2, 20/07; previously this was inferred from `Array.isArray(state.data[key])` at call time, which depended on how/when that key had been populated rather than on what the list actually is).
4. **Global mutable `state` object** — holds the MSAL account, resolved `role`, current user (`me`), SharePoint `siteId`/`listIds`, and `state.data.{operatori,utenti,progetti,sessioni,chiusure,impostazioni,report}`. There's no framework/store; UI code reads/writes `state` directly and re-renders by calling the relevant `renderX()` function.
5. **Auth + role resolution** (`initMsal`, `doLogin`, `doLogout`, `getToken`, `resolveRole`) — role is derived from Entra ID app roles claim, or by matching the signed-in email against the `operatori` list, or against `CFG.adminEmails`. If no operators exist yet, the first login bootstraps as Admin (`state.bootstrap`). Two roles: `Admin` (full access) and `Operatore` (sees only their own calendar/sessions/availability).
6. **Navigation** (`TABS`, `buildNav`, `showTab`) — each tab maps to a `<section id="view-*">` and triggers a `renderX()` call; there's no router, just show/hide of sections. A `TABS` entry may carry a `children` array instead of being a direct leaf: `buildNav()` renders it as a dropdown (`.nav-drop`/`.nav-drop-toggle`/`.nav-drop-menu`) opened by hover on pointer devices and by click/tap on the parent button otherwise (`chiudiNavDrop()`, a document-level click listener) — needed since touch has no hover. Only Admin's "⚙️ Impostazioni" entry does this today (children: Impostazioni itself + "📅 Chiusure", Ciclo H, 21/07) — Chiusure has no top-level nav entry of its own anymore, only reachable through that dropdown. `trovaTabById(role,id)` looks up a tab (leaf or nested child) by id; `showTab()` marks the parent toggle `.active` too whenever the currently shown id is one of its children, so the dropdown's origin stays visible.
7. **Rendering** — plain DOM/innerHTML templating per view (`renderCalendario`, `renderSessioni`, `renderOperatori`, `renderUtenti`, `renderProgetti`, `renderDisponibilita`, `renderChiusure`, `renderImpostazioni`, ...), all string-built HTML with the `esc()` helper for escaping. Modals are generic: `openModal(html, onCancel, isDirty, onSaveTrigger)`/`openModalStacked(html, onCancel, isDirty, onSaveTrigger)` inject markup into `#modal-root`; `onCancel` restores state on abandonment (e.g. the disponibilità snapshot), `isDirty`/`onSaveTrigger` power the unsaved-changes guard (see "Regole UX" below). A single Escape-handler stack (`pushEsc`/`popEsc`) keeps nested modal/popup/confirm-dialog layers from reacting to the same keypress.
8. **Scheduling engine** — the core domain logic, structured as **three passes** (the AI-assisted path only runs the first two — see below), shared by both generation paths via common helpers (`determinaAmbito`, `sessioniDaConservare`, `sedePriorita`, `minutiSettimana`, `risolviOnlineDaCasa`, `decidiOnlineDaCasa`, plus `riepilogoSettimanaleDaSessioni`, `costruisciReportGenerazione`, `calcolaMetricheReport` for the report):
   - **Scope ("ambito") and protected sessions**: `determinaAmbito(spid, nameFilter)` resolves which projects are in scope for a given generation run (single project / name filter / all active), independent of remaining monte-ore. `sessioniDaConservare(ms, scopeIds)` returns every session that must survive the run untouched — i.e. everything **except** `proposta` sessions belonging to in-scope projects. `confermata`, `eseguita`, `assenza ingiustificata`, `annullata`, and any `proposta` from an out-of-scope project are never deleted/recreated. All of these (except `annullata`, which is treated as a freed slot) remain **active constraints**: they pre-populate the busy-operator/busy-room trackers before any new placement happens.
   - **Passata 1 — placement** (`generateMonth()` inline; `generateMonthAI()`'s IA-response validation loop): assigns sessions for in-scope projects given frequency/duration, operator availability (`effRng`), room constraints (Cesate has 6 rooms, Busto Arsizio has 2, same room per half-day), a genuinely-enforced **5-minute gap** between consecutive sessions (operator, room, and same client across their own other projects — `rfreeConGap()`/`GAP_MINUTI`; fixed in Ciclo F.1, 21/07: before this, `generateMonth`'s operator check was two redundant lines that canceled each other out, so the margin was declared but never actually applied — see CONTESTO.md backlog #13, now closed), in-day travel time between sites, and weekly-hour contract caps — but does **not** yet decide whether an `Online` session happens from home or on-site (that field, `onlineDaCasa`, is left `null`). Project order: primary key is the rigidity index (`calcStrettezza`); secondary key at a tie is `sedePriorita()` (in-presence < composite Cesate+Online/Domicilio < pure remote). Operator choice within an eligible pool is pre-sorted by `minutiSettimana()` so that `Assunto` operators still under their weekly contracted hours are tried before everyone else (P.IVA collaborators fill the rest). In `generateMonthAI()`, Passata 1 also rejects any AI-proposed session that overlaps a protected/out-of-scope session's operator or room. **Frequenza a tre modalità** (`modalitaFrequenza(p)`, Ciclo F.1, 21/07 — see the S11 bullet under "Modifiche dal campo" for full detail): `settimanale` (default, unchanged — a count per fixed 7-day window, `bucketSettimana`/`maxNuoveSettimana`) or, for `quindicinale`/`mensile`, a single candidate day for the whole month computed as a distance from the project's last valid session (`ultimaLezioneValida`), with Sunday/closure slittamento (`slittaGiornoValido`) — this branch is only in `generateMonth`, not yet in Passata 3 or `generateMonthAI` (Ciclo F.2, not yet done).
   - **Passata 2 — online resolution** (`risolviOnlineDaCasa(ms, newS, keep, anom)`): runs after all placement is done (both after Passata 1 in the AI path, and again after Passata 3 in the algorithmic path — see below), over the *combined* set of new + kept sessions. For each operator+day it sorts **every** session chronologically (including sessions from projects outside this run's scope) and calls `decidiOnlineDaCasa()` per `Online` session in `newS`, applying the travel-time rule (`tempoCasa` if the nearest in-presence session is at Cesate; if at Busto Arsizio, `tempoBustoOperatore()` — `tempoBustoCesate` if the operator also has a Cesate presence that same day, else `tempoBustoCasa`, since Ciclo B/S8, 17/07) and treating an already-resolved "online in sede" session as an extended physical presence for later decisions the same day. Never mutates sessions outside `newS` (protected sessions' own `onlineDaCasa`/`aula` are left as-is). The optional `anom` parameter collects a one-per-operator warning when `tempoBustoCesate` is still unset and the fallback to `tempoBustoCasa` had to be used.
   - **Passata 3 — riparazione completezza** (`generateMonth()` only, right after Passata 1 and before the final Passata 2 re-run; **not present in `generateMonthAI()`**): for each weekly bucket (`riepilogoSettimanaleDaSessioni`, 7-day windows within the month) where a project's `piazzate` (sessions placed) fall short of `richieste` (its weekly `frequenza`) due to a real availability/room shortage, tries to close the gap by moving/swapping **only `proposta` sessions belonging to in-scope projects** — confirmed, eseguita, assenza ingiustificata, annullata, and any out-of-scope `proposta` are never touched. It runs in two ordered phases per shortage: first trying moves using only `Assunto` operators from the project's `operatoriAmmessi` pool, then falling back to the full admitted pool. It attempts **only operator reassignment, never a room swap** — if every room is full at a candidate slot it skips that slot (`aule piene: la Passata 3 non tenta scambi di aula, solo di operatore`). When a candidate slot is already occupied by exactly one other session, it only swaps if that blocking session is itself a movable in-scope `proposta` and an alternative slot for it can be found (`trovaAlternativaBlocker`); if the blocker is `confermata`/`eseguita`/out-of-scope/otherwise not movable, or no alternative slot exists, the move is **not executed** — a human-readable suggestion is pushed to `suggerimentiFuoriAmbito` instead (surfaced in the report), never applied automatically. The whole pass is capped at **`MAX_MOSSE = 40`** attempted moves total across all shortages. After Passata 3, Passata 2 (`risolviOnlineDaCasa`) is re-run over the updated new+kept set to resolve `onlineDaCasa` for whatever it moved or added. **Vincolo duro sulla frequenza settimanale** (S7, Ciclo C, 17/07): `piazzate` in ogni riepilogo settimanale è il **totale** (sessioni esistenti/kept del progetto in quella finestra + le nuove piazzate in questo run), non solo le nuove — Passata 1 non piazza mai più di `frequenza - esistenti` sessioni nuove per finestra (`maxNuoveSettimana()`), quindi `missing = richieste - piazzate` in Passata 3 riflette già il vero margine residuo e la riparazione non può mai portare il totale oltre `frequenza`, nemmeno spostando/scambiando sessioni. Lo stesso vincolo è applicato anche nella validazione post-IA di `generateMonthAI()` (sessione scartata se la sua settimana ha già raggiunto la frequenza del progetto).
   - **Report completo di generazione** (`costruisciReportGenerazione`, one "📄 Report completo" button after every run, both entry points): for each user/week/project, richieste vs piazzate; when short, the diagnosed cause(s) (`diagNoUtente` — insufficient client availability, listing the exact weekdays; `diagAulaPiena` — rooms full; `diagNessunOperatore` — no operator available in the slot); actionable suggestions (`suggerimentiRichieste`, e.g. "ask this client for availability on these days", "consider widening/adding operators to this project's pool", "rooms saturated, consider other times/online") plus, algorithmic path only, the Passata 3 move log (`mosseRiparazione`) and untouched-blocker suggestions (`suggerimentiFuoriAmbito`); and month-wide metrics from `calcolaMetricheReport`: **buchi operatori** (idle minutes between in-presence sessions on days the operator is physically on-site), **% online da casa** (share of resolved `Online` sessions actually done from home), **saturazione aule** (per-day room occupancy % over the 09:00–19:30 window), **carico Assunti vs P.IVA** (total scheduled hours by contract type).
   - Both entry points end by first **deleting from SharePoint** the sessions being replaced (`proposteDaSostituire(state.data.sessioni, ms, scopeIds)` — the exact complement of `sessioniDaConservare`, i.e. the in-scope `proposta` sessions of that month) via `deleteRecord`, only then setting `state.data.sessioni` to the preserved set and persisting `newS` one by one via `saveRecord('sessioni', ...)`. **Fixed in Ciclo E.1 (20/07) — bug history, verified against git history (not just assumed)**: before this fix, replacement only reassigned the in-memory `state.data.sessioni` array; the old proposals' SharePoint list items were never deleted, so they resurfaced as duplicates on the next page load. The repo did briefly use `localStorage` before SharePoint (first `index.html` commit, 07/07, vs. the switch to Graph the very next day, 08/07) — but `generateMonth` didn't exist yet in either of those commits; it was written later, once the app was already on SharePoint. So this isn't literal leftover pre-SharePoint code, but a mental-model mismatch ("replace the list" thinking, appropriate for a single-in-memory-object persistence model) that outlived the brief period it fit, never reconciled with SharePoint's per-record persistence when the generator was written (vedi `CONTESTO.md`/`VERIFICA.md` per il dettaglio della verifica). If a deletion fails, the whole generation aborts before saving anything new — this prevents ending up with both the not-yet-deleted old proposals and freshly created duplicates.
   - "🗑 Svuota proposte del mese" mirrors the same scope rule (now sharing `proposteDaSostituire` with the two generation paths instead of duplicating the filter): it only deletes `proposta` sessions of in-scope projects, never `confermata`/`eseguita`/`assenza ingiustificata`/`annullata` or out-of-scope proposals.
   - **`generateMonthAI()` limitations** (documented so the two paths aren't assumed equivalent): it does **not** implement `tipiSessione` rotation (session-type components are simply not generated/considered) and does **not** run Passata 3 (no repair pass — its report only reports the shortfall, with no `mosseRiparazione`/`suggerimentiFuoriAmbito`, since — as the code comment puts it — "qui il piazzamento è deciso dal modello, non da una ricerca deterministica su cui intervenire con scambi"). The composite `Cesate+Online`/`Cesate+Domicilio` site the model proposes is validated against the allowed pair and the whole session **discarded** if invalid (not corrected to a valid site); the `onlineDaCasa`/room-in-sede decision, on the other hand, is always the same deterministic Passata 2 (`risolviOnlineDaCasa`) run once over the AI's output after the response comes back — never left to the model.
9. **AI chat assistant** (`sendChat`, `bCtx`) — also calls `CFG.claudeProxy` with a context string built from current operators/utenti/progetti counts, purely for Q&A (not persisted beyond the in-memory `chatH` array).
10. **Excel import/export** (`handleImport`, uses the `xlsx` SheetJS library loaded from a CDN) for bulk-importing operators/clients from `.xlsx` templates.
11. **Comunicazioni** (Calendario tab, "Invio calendario settimanale" panel) — `mkWeekMsg(uteId)` builds a plain-text weekly appointment summary (next Mon–Sun, excluding `annullata` sessions) for one client, sent via `#btn-wa` (`wa.me` deep link) or `#btn-mail` (`mailto:`). **Nota di fedeltà**: il pulsante "📤 Invio mese" nella toolbar del Calendario (id `cal-send-month`, title "Esporta messaggi mese") è presente in markup e reso visibile solo per Admin, ma **non ha alcun listener collegato** nel codice attuale — nessuna funzione di export mensile è implementata; è un elemento UI non funzionante (vedi `VERIFICA.md`, sezione Discrepanze).

### Key domain concepts worth knowing before editing scheduling/session code

- **Sedi** (sites): `Cesate`, `Busto Arsizio`, `Online`, `Domicilio`, plus composite modes `Cesate+Online` / `Cesate+Domicilio` (rinominate dal 17/07, Ciclo A — erano `Presenza+Online`/`Presenza+Domicilio`; vedi `CONTESTO.md`, Registro delle decisioni, per data e motivazione) that resolve to a concrete site per-session based on operator/room availability. **Business rule**: for `Cesate+Online` the effective site can only resolve to `Cesate` or `Online`; for `Cesate+Domicilio` only to `Cesate` or `Domicilio` — **never** `Busto Arsizio` (Busto Arsizio has no online sessions). This is enforced in `generateMonth()` (slot-compatibility filter and effective-`sede` resolution) and in `generateMonthAI()` (prompt constraint **and** a hard post-validation that discards any AI-proposed session violating it) — keep both in sync if this rule ever changes.
- **Aule** (rooms): fixed lists `AULE_CESATE` (6) and `AULE_BUSTO` (2); an operator keeps one room for the whole day/half-day once assigned (`opRoom` map keyed by `operatorId|date` or `operatorId|date|halfDay`). An `Online` session resolved as "in sede" (see Passata 2 above) still occupies a room like an in-presence one — it does **not** free the room.
- **Orari e giorni**: the center's bookable window is **09:00–19:30**, enforced everywhere a slot is computed (`tmin('09:00')`/`tmin('19:30')` clamps in Passata 1, Passata 3, and the AI prompt's "VINCOLI INDEROGABILI"). **Sunday is always excluded from generation** — every day-loop in `generateMonth`/Passata 3 skips `dn==='Dom'` outright, and the AI prompt states "Domenica: centro chiuso".
- **Disponibilità** (availability): weekly recurring slots plus per-date exceptions (`eccezioni`/`availOverrides`), read through `effRng()`, which supports both a newer array-of-exception-objects format and a legacy per-date object format — check both when touching availability logic. Each weekly/exception slot can carry a `sedi` list (`sdComp`/the site-compatibility check around line 1541): a slot tagged only `Cesate` does **not** implicitly cover `Online` (or vice versa) — for a composite `Cesate+Online`/`Cesate+Domicilio` project the slot must explicitly list `Cesate` **or** the remote site to match. **S5 (Ciclo C, 17/07)**: in the project editor (weekly `pe-av`, daily exception fasce, and "Applica a più giorni"), the sede tags offered for a slot are restricted to `sediAmmesseProgetto(p.sede)` — the project's own `Sede` (single site → that tag; composite → Cesate + the remote one). A tag already selected on an existing slot is never hidden/force-unchecked even if it falls outside the newly-allowed set after a `Sede` change mid-edit — it stays visible, checked, and visually flagged (`sede-extra` class) alongside an explicit warning banner listing the affected days, so nothing is silently dropped.
- **Assenze operatore** (via `eccezioni`, read by `effRng`): `malattia` always blocks the **entire day**, no time-band option. `ferie`/`permesso_visita`/`permesso_studio` take an **optional** time band (`da`/`a`): with no band, the whole day is unavailable; with a band, only that band is subtracted from the operator's own declared weekly availability for that day (`subtractWindow(weekly(), da, a)`) — the rest of the day stays available **only inside the operator's already-declared weekly availability slots for that day, never beyond them** (the exception can shrink availability, it can never grant more than what's on file).
- **Requisito formazioni**: when a project has `tipiSessione`, an operator is only eligible for a given rotation slot if they hold **every** training (`formazione`) required by that session type's `componenti` (`reqForms.every(rf=>opForms.includes(rf))`, Passata 1 and Passata 3) — a partial match is not enough.
- **Tipi di sessione** (session-type rotation): when a project has `tipiSessione`, the rotation index (`sessionCountByProject`, keyed per project) advances continuously across the **entire month** and is never reset at the start of a week — including for sessions recovered by Passata 3, which keep incrementing the same counter. **Approccio A**: one single operator covers the whole session (all of that session type's method "componenti"), never split across multiple operators.
- **Progetti** (projects) carry `monteOre` (hour budget) — projects flagged `noMonteOre` (manual/no-hour-budget scheduling) are **excluded from automatic generation entirely** (filtered out of scope at both `determinaAmbito` call sites) and must be calendared by hand — `modalitaFrequenza` (`settimanale`/`quindicinale`/`mensile`, Ciclo F.1, 21/07 — see "Modifiche dal campo", S11), `frequenza` (sessions/week, used only in `settimanale` mode; `null` for the other two), `durataSessione`, `operatoriAmmessi` (eligible operators), an optional `operatoreFisso` flag (already implemented: when set, placement only ever tries the *first* operator in `operatoriAmmessi`, never rotating to others), and optionally `tipiSessione` (session-type rotations with per-method duration "componenti").
- **Sessioni** states: `proposta`, `confermata`, `eseguita`, `assenza ingiustificata`, `annullata` (see `STATI_SESS`). Only `proposta` sessions of in-scope projects are ever deleted/recreated by a generation run or by "Svuota proposte del mese" — every other state is a protected, active scheduling constraint (see Scheduling engine above). `oreErog()` (monte-ore consumed) counts `eseguita` and `assenza ingiustificata`, excludes `annullata` (which stays recoverable) and `proposta`/`confermata` (not yet consumed). Sessions from **different projects of the same client never overlap**: every placement/repair pass collects busy-blocks across *all* of that client's projects (`allPU`/`uBN`, not just the project being scheduled) and requires the candidate slot to be free against that combined set before placing.
- **Chiusure**: center-wide closures (holidays/vacation) that block session generation on those dates; Italian public holidays (including computed Easter/Easter Monday) are precomputed via `getFestivita`/`getPasqua`.

When making changes, keep everything self-contained in `index.html` (no framework or bundler is set up), preserve the JSON-blob-in-SharePoint-list persistence model, and be careful with the two scheduling code paths (`generateMonth` vs `generateMonthAI`) since they duplicate validation/session-shape logic independently.

## Regole UX

- **Modifiche non salvate nelle schede modali**: ogni scheda modale con form (operatori, utenti, progetti, sessioni, disponibilità — popup giorno e "Applica a più giorni") confronta lo stato del form con quello di apertura. Se l'utente prova a chiuderla (click fuori/backdrop, Esc, o pulsante "Annulla"/di chiusura) con modifiche non salvate, appare un dialogo con tre scelte: **Salva ed esci**, **Esci senza salvare**, **Continua a modificare**. Nessuna via di chiusura deve mai perdere dati non salvati senza avviso. Implementato tramite i parametri `isDirty`/`onSaveTrigger` di `openModal`/`openModalStacked` e la funzione condivisa `confirmUnsavedChanges()`.
- Questo confronto esclude le eccezioni di disponibilità (`eccezioni`) quando fanno parte anche del form principale (operatori/progetti): quelle sono salvate immediatamente su ogni click-giorno/"Applica a più giorni" e già gestite dal ripristino via snapshot descritto sopra — non devono duplicare l'avviso.

## Regole di business pianificate (non ancora implementate)

- **Pausa pranzo** (da implementare in un prossimo ciclo): ogni operatore deve avere una pausa pranzo di 60 minuti consecutivi interamente dentro la finestra 12:00–14:30 (ultima fascia valida 13:30–14:30). La pausa è implicita (non richiede uno slot vuoto esplicito) se l'operatore inizia a lavorare alle 13:30 o dopo, oppure se finisce entro le 13:30.
  - **Senza viaggio** (operatore già in sede tutta la giornata, oppure tutte le sessioni online): pausa tassativa di 60 minuti pieni, nessuna riduzione ammessa.
  - **Con viaggio casa→sede dopo la pausa**: tolleranza solo in difetto, la pausa può scendere fino a un minimo di 50 minuti (riduzione massima 10 minuti), sempre interamente dentro la finestra, e solo se serve a piazzare sessioni. L'algoritmo preferisce sempre i 60 minuti pieni e riduce solo se necessario.
  - Il viaggio casa→sede avviene **dopo** la pausa e può estendersi oltre le 14:30 senza limite. Esempio 1: pausa 13:30–14:30, viaggio 14:30–15:00, prima sessione alle 15:00. Esempio 2 (tolleranza con viaggio): pausa 13:30–14:20, viaggio 14:20–14:50, prima sessione alle 14:50.
  - La Passata 2 (`risolviOnlineDaCasa`) può scegliere "online in sede" per far quadrare la pausa (l'aula resta occupata in quel caso); in tal caso l'operatore è considerato già in sede (il viaggio è stato fatto al mattino prima di iniziare) e la pausa torna tassativa di 60 minuti.
  - Il report di generazione può suggerire riduzioni oltre i limiti sopra o spostamenti della pausa fuori finestra per sbloccare sessioni, ma non deve mai eseguirli in automatico.

- **Nuovo stato di sessione "calendarizzata"** (da implementare insieme alla riparazione interattiva sotto — vedi `CONTESTO.md`, Registro delle decisioni, voce 8, per data e motivazione): la generazione produrrà le sessioni direttamente nello stato **calendarizzata**, non più `proposta` come oggi. La Passata 3 diventerà **a convergenza** — giri di riparazione illimitati finché un giro non migliora più, al posto dell'attuale tetto `MAX_MOSSE=40` — e in questa modalità potrà modificare liberamente **solo** le sessioni calendarizzate. Quando un calendario viene inviato a una famiglia (funzione di invio, vedi "Comunicazioni" sopra e il backlog "Ciclo comunicazioni" in `CONTESTO.md`), le sessioni coinvolte passano **automaticamente** da calendarizzata a `proposta`; da quel momento l'algoritmo non le modifica più da solo — `proposta` e `confermata` possono essere coinvolte solo tramite la riparazione interattiva sotto o i suggerimenti del report. `eseguita`, `assenza ingiustificata` e `annullata` restano, come oggi, sempre intoccabili. **Nota di revisione**: oggi il codice crea direttamente lo stato `proposta` (non esiste ancora `calendarizzata`); la catena degli stati va rivista in `calendarizzata → proposta → confermata → esiti` (`eseguita`/`assenza ingiustificata`/`annullata`), e il passaggio automatico calendarizzata→proposta va agganciato alla funzione di invio del calendario.

- **Riparazione interattiva a due fasi di domande** (cantiere dedicato dopo il collaudo automatico — vedi `CONTESTO.md`, Registro delle decisioni, voce 9, per data e motivazione): quando la Passata 3 a convergenza (sopra) non riesce a chiudere una carenza da sola, pone domande interattive invece di limitarsi a un suggerimento nel report.
  - **Fase domande 1 (entro le regole)**: domande su spostamenti, scambi, cambi di orario, cambi di aula, cambi online/presenza che coinvolgono sessioni `proposta` o `confermata`, sempre rispettando i parametri del progetto e le disponibilità dichiarate di operatori e utenti.
  - **Fase domande 2 (eccezioni)**, esempi non esaustivi: operatore non assegnato al progetto ma formato e disponibile; orario fuori dalle disponibilità dichiarate di utente o operatore; riduzione della pausa pranzo oltre i limiti tollerati; sforamento delle ore contrattuali di un Assunto.
  - Ogni domanda ha tre pulsanti: **Applica** (la modifica viene eseguita e la riparazione riparte), **Attesa** (domanda sospesa in un pannello di riepilogo dedicato, dove la risposta può essere cambiata in seguito quando utente/operatore avranno risposto fuori dall'app — le domande in attesa vanno salvate su SharePoint per sopravvivere alla chiusura del browser), **Ignora** (la proposta viene rifiutata ma registrata nel report di generazione per poterla ricontrollare).
  - La Fase 2 non inizia finché tutte le domande della Fase 1 non hanno risposta Applica o Ignora — le domande lasciate in Attesa bloccano il passaggio alla Fase 2 finché non vengono risolte in un modo o nell'altro.

- **Pseudonimizzazione dati verso l'IA — GDPR/sicurezza** (priorità alta, da implementare in un prossimo ciclo): qualsiasi dato inviato a `CFG.claudeProxy` (sia l'assistente IA `sendChat`/`bCtx`, sia il generatore IA `generateMonthAI`) dovrà essere pseudonimizzato — mai nomi, cognomi o altri dati identificativi reali di utenti/operatori, e mai i loro contatti (telefono, email) né le credenziali (`credenziali[].password`) salvate in anagrafica. **Fino all'implementazione, ogni modifica alle funzioni che costruiscono il payload per `CFG.claudeProxy` deve tener conto di questo vincolo** (non aggiungere nuovi campi identificativi reali al prompt/contesto nel frattempo).

  - **Punto di partenza del conteggio** (`ultimaLezioneValida(sessioni,progettoId)`): l'ultima sessione in stato `proposta`/`confermata`/`eseguita` del progetto — `annullata`/`assenza ingiustificata` non contano, si prende la valida precedente. Cercata su **tutto** lo storico (non solo il mese in generazione, come richiesto esplicitamente dal vincolo tecnico cross-mese sopra): in `generateMonth` viene interrogata su `keep` (che già esclude solo le "proposta" di questo mese in ambito che il run sta per sostituire — non l'intero `state.data.sessioni` grezzo, per evitare che una propria proposta non ancora sostituita da questo stesso run alteri il calcolo).
  - **Candidata e slittamento** (`addGiorni`, `slittaGiornoValido`, `finestraOk`, `giorniTraDate`): la data candidata è ultima-lezione-valida + il **minimo** della finestra (12 per quindicinale, 23 per mensile — non il punto medio: nessuna indicazione esplicita su quale giorno preferire entro la finestra, scelto il più immediato), poi slittata al primo giorno che non sia domenica né una chiusura centro. Lo slittamento ha **priorità sulla finestra**: può produrre una distanza fuori dai limiti dichiarati (segnalata in `anom`), ma non salta mai la lezione.
  - **Un solo giorno candidato per mese per progetto** (limite dichiarato di questa prima tappa): se operatore/utente non sono disponibili esattamente quel giorno, il progetto risulta non piazzato in questo run (segnalato in `anom`/report) — nessuna ricerca automatica di un giorno alternativo altrove nella finestra, a differenza della modalità settimanale (che esplora tutti i 7 giorni della sua finestra). Possibile affinamento per un ciclo successivo.
  - **Niente sconfinamento cross-mese** (esplicitamente fuori scopo di F.1): se la candidata calcolata cade in un mese diverso da quello generato, `generateMonth` non piazza nulla in questo run. Se cade **prima** del mese generato, segnala un'anomalia esplicita ("lezione scaduta", verifica manuale consigliata); se cade **dopo** (non ancora dovuta questo mese), nessuna azione né avviso — è il caso normale per la maggior parte dei mesi con cadenze superiori alla settimana.
  - **Ordinamento priorità** (`calcStrettezza`, mode-aware da F.1): per queste due modalità usa una frequenza settimanale equivalente nominale (7 / punto medio della finestra) solo per ordinare i progetti in Passata 1 a parità di altre condizioni — non introduce alcun vincolo di piazzamento nuovo, la formula per la modalità settimanale resta invariata.

## Avvertenze di sicurezza esistenti (non pianificate, solo documentate)

- **Credenziali piattaforme in chiaro**: il campo `credenziali` (array `{piattaforma, uid, password}`) di operatori e utenti (`buildCredUI`, righe ~2056+) viene serializzato tale e quale in `saveRecord` (`Data: JSON.stringify(payload)`) e salvato **in chiaro** nelle liste SharePoint — nessuna cifratura lato client o server. Uso accettato solo per credenziali di piattaforme di lavoro (es. accesso a portali BrainRx); **non deve mai essere usato per credenziali critiche o bancarie**. ## Manutenzione del CONTESTO.md

`CONTESTO.md`, nella root del repo, è il documento di contesto del progetto usato nelle conversazioni di consulenza con Claude (fuori da Claude Code). Regola permanente:

- Alla fine di ogni sessione di lavoro che produce modifiche al codice, **prima del commit finale**, aggiorna `CONTESTO.md`:

  - aggiorna la sezione "Backlog" se sono emerse nuove voci o se alcune sono state completate (rimuovile o segnale come fatte);
  - aggiorna la sezione "Registro delle decisioni" se in sessione sono state prese nuove decisioni non puramente tecniche (vedi Prassi di chiusura ciclo sotto).
- `CONTESTO.md` va sempre incluso nello stesso commit delle modifiche di codice a cui si riferisce, mai in un commit separato successivo.
- **Conteggio righe di `index.html`**: nei documenti (questo file, `CONTESTO.md`, `VERIFICA.md`) non si scrive mai un conteggio esatto di righe del codice, solo un valore indicativo (es. "~2300 righe"). Il conteggio esatto (`wc -l index.html`) si verifica solo durante gli audit, per non generare una discrepanza documentale a ogni piccola modifica al codice.

## Prassi di chiusura ciclo

Quattro regole permanenti, valide per ogni sessione di lavoro (anche quelle solo documentali, senza modifiche a `index.html`):

**(a) Chiusura standard**: ogni sessione di lavoro si chiude con una verifica automatica documentata in `VERIFICA.md`, il conseguente aggiornamento di `CONTESTO.md` (vedi sezione sopra) e un commit (con push) con messaggio descrittivo in italiano.

**(b) Registro di sessione**: ogni voce di `VERIFICA.md` deve includere una sezione "Registro di sessione" che riporta — a beneficio di un consulente esterno che verifica dal solo GitHub, senza accesso alla chat — le istruzioni date da Simone direttamente in sessione oltre al prompt iniziale, le domande poste a Simone e le risposte ricevute, e le decisioni prese di conseguenza. Nessuna istruzione, domanda/risposta o decisione presa a voce/in sessione deve andare persa: se non è nel prompt iniziale ma ha influenzato il lavoro, va registrata qui.

**(c) Verifica multi-passata a quattro fonti**: ogni verifica richiesta da Simone va condotta con più passate di analisi, ad angoli diversi e complementari (es. una passata per punto/regola, una per coerenza interna di ciascun file, una di confronto incrociato fra file, una finale di rilettura completa) — senza numero di passate prefissato: **minimo 4**, e si continua finché una passata aggiuntiva non trova più nulla di nuovo. Ogni verifica deve incrociare le **quattro fonti**: `CLAUDE.md`, `CONTESTO.md`, `VERIFICA.md` e il codice reale di `index.html`. Per ogni informazione va controllato che sia presente in ogni fonte in cui deve stare, e che le fonti siano coerenti fra loro; ogni incongruenza trovata va **segnalata, mai corretta in silenzio** (le discrepanze fra queste regole e il codice reale vanno in fondo a `VERIFICA.md`, sezione "Discrepanze da discutere" — senza toccare il codice per farle sparire). Le informazioni e le decisioni non devono mai perdersi per strada.

**(d) Igiene del contesto**: a ciclo chiuso (commit + push + registro di sessione aggiornato) Simone lancia `/clear` e riparte con il prompt zero di allineamento. Mai `/clear` a metà ciclo con decisioni non ancora scritte nei file: in quel caso usare `/compact` oppure chiudere prima il ciclo con un commit documentale.

**(e) Data di completamento = data del commit** (20/07/2026): in Cronologia e Backlog (`CONTESTO.md`), la data di completamento di un ciclo è sempre quella del commit che lo chiude, non la data in cui Simone ha definito la specifica (che può essere precedente — vedi Registro delle decisioni, voce 39, per il caso che ha reso necessaria questa regola).
